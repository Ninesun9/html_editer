# HtmlFox Project Design

## 1. Product positioning

HtmlFox is a lightweight desktop tool for **editing AI-generated HTML files locally**.

It is designed for users who:

- already have an HTML page generated by AI
- want to edit text, links, images, and layout quickly
- do not want to work only in raw code

This project is **not** positioned as:

- a full IDE
- a low-code page builder
- a Figma-style drag-and-drop editor

The product goal is clear:

**help users open an HTML file, edit it faster, preview changes instantly, and save it safely**

## 2. Core user value

The editor should provide four core values:

1. **Open and use immediately**  
   Users can load a local `.html` file and start editing without project setup.

2. **Edit with confidence**  
   Users can modify HTML in a proper editor with syntax highlighting and formatting.

3. **See results instantly**  
   The preview updates in real time so users can understand changes visually.

4. **Prepare for visual editing**  
   The architecture must support the future ability to click an element in preview and jump to the matching source.

## 3. MVP scope

## 3.1 In scope

The first version focuses on the edit-preview-save loop:

- open local HTML files
- edit HTML with Monaco
- live preview in an iframe
- save and save as
- format HTML with Prettier
- switch preview between desktop and mobile widths
- keep a recent files list
- start from a built-in sample HTML document

## 3.2 Out of scope

These are intentionally postponed:

- DOM-to-source mapping
- inline text editing inside preview
- visual attribute inspector
- drag-and-drop layout editing
- multi-page project management
- advanced CSS tooling
- JS debugging
- AI chat-driven editing

## 4. Target tech stack

To prioritize compatibility and stable desktop delivery:

- **Desktop shell:** Electron
- **UI framework:** Vue 3
- **Language:** TypeScript
- **State management:** Pinia
- **Code editor:** Monaco Editor
- **Formatting:** Prettier
- **Build tooling:** electron-vite

## 5. Why this stack

### Electron

- strong Windows compatibility
- mature packaging and runtime behavior
- proven ecosystem for desktop editors

### Vue 3 + TypeScript

- fast to build UI with
- strong maintainability
- simple state flow for a tool-like application

### Monaco

- VS Code-like editing experience
- good HTML editing support
- future-ready for source range highlighting

## 6. High-level architecture

The application is split into three layers:

### 6.1 Electron main process

Responsibilities:

- create the application window
- open file picker dialogs
- save files
- persist recent files
- expose secure IPC commands

Main files:

- `electron\main\index.ts`
- `electron\preload\index.ts`

### 6.2 Renderer application

Responsibilities:

- render the app shell
- manage document state
- host Monaco editor
- render preview
- coordinate file actions and formatting

Main files:

- `src\App.vue`
- `src\components\*.vue`
- `src\stores\document.ts`

### 6.3 Shared service layer

Responsibilities:

- HTML formatting
- Monaco worker setup
- default starter HTML
- later: parsing, mapping, and serialization

Main files:

- `src\services\formatHtml.ts`
- `src\services\monaco.ts`
- `src\services\defaultHtml.ts`

## 8. Main UI layout

The UI uses a three-column layout:

1. **Left sidebar**
   - current file
   - recent files

2. **Center editor pane**
   - Monaco HTML editor
   - editing status

3. **Right preview pane**
   - iframe preview
   - desktop/mobile viewport mode

Top toolbar actions:

- open
- save
- save as
- format
- device mode switch

## 9. Current state model

The first version keeps state in a single Pinia document store.

Key state:

- `filePath`
- `html`
- `dirty`
- `recentFiles`
- `deviceMode`
- `statusMessage`
- `errorMessage`

This is enough for the first version and can later be split into:

- document state
- selection state
- preview state
- history state

## 10. Current workflow

### 10.1 Open file

1. Renderer calls preload API
2. Main process opens system file picker
3. Main process reads HTML file
4. Renderer updates store
5. Preview refreshes
6. Recent files list updates

### 10.2 Edit content

1. User edits HTML in Monaco
2. Store updates HTML content
3. Preview refresh is debounced
4. iframe rerenders current HTML

### 10.3 Save content

1. Renderer sends HTML to main process
2. Main process writes file
3. Store clears dirty state
4. Recent files list updates

### 10.4 Format content

1. Renderer sends current HTML to Prettier
2. Formatted HTML replaces current content
3. Preview updates immediately

## 11. Security and compatibility decisions

The first version follows these rules:

- use `contextIsolation: true`
- keep `nodeIntegration: false`
- expose only required file APIs through preload
- render preview inside iframe
- separate main and renderer responsibilities

Compatibility-oriented decisions:

- Electron instead of Tauri
- Windows-friendly path handling
- Monaco worker setup through Vite worker imports

## 12. Known limitations in the current version

The first version is intentionally incomplete:

- preview elements are not yet mapped back to source code
- there is no inline text editing in preview
- there is no undo/redo history manager yet
- there is no structured DOM tree yet
- there is no visual attribute inspector yet

These are planned next, but the current foundation is designed to support them.

## 13. Next milestone design

The next milestone is the **DOM-to-source mapping layer**.

### Goal

Allow users to click an element in preview and jump to its matching HTML source.

### Planned approach

1. Parse HTML into an AST
2. Inject stable `data-node-id` markers
3. Build `nodeId -> sourceRange` mappings
4. Let preview emit clicked `nodeId`
5. Make Monaco reveal the matching range

### New modules expected

- `parser.service.ts`
- `mapping.service.ts`
- `selection.store.ts`
- `preview-bridge.ts`

## 14. Development roadmap

### Phase 1: foundation

- Electron app shell
- Vue renderer
- Monaco integration
- preview rendering
- file operations

### Phase 2: source mapping

- HTML parsing
- node identifiers
- preview click to source jump
- source selection to preview highlight

### Phase 3: visual editing

- inline text edit
- attribute editing
- selected node inspector

### Phase 4: usability

- undo/redo
- snapshots
- unsaved-change prompts
- error handling polish
- larger file performance tuning

## 15. Success criteria for version 1

The first version is successful if a user can:

1. launch the app
2. open an AI-generated HTML file
3. edit the HTML comfortably
4. see live visual feedback
5. format the document
6. save the result locally without breaking the workflow

## 16. Summary

This project should stay focused on one thing:

**make AI-generated HTML easier to modify**

The current implementation already establishes the right first-version direction:

- compatible desktop stack
- clean renderer/main split
- solid editing foundation
- architecture ready for later visual mapping features
